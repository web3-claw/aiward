# Local Install and Usage

This guide explains how to install EnvGate from this local checkout and test the
passive MVP on your machine.

EnvGate is currently passive and explicit-call only. It does not hook your shell,
scan all terminal input, or monitor commands that bypass it. Secrets are protected
when `.env` is locked by EnvGate and secret-bearing commands run through
`envgate run`.

## Prerequisites

Install Rust with Cargo:

```bash
rustc --version
cargo --version
```

If either command is missing, install Rust from `rustup`:

```bash
curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh
source "$HOME/.cargo/env"
```

## Verify the Checkout

From the EnvGate repository:

```bash
cd /Users/eduardonicoleit/Documents/env-gate

cargo fmt --check

cargo test -- --test-threads=1
```

To run the full coverage gate:

## Run Without Installing

You can test the CLI directly from the checkout:

```bash
cargo run -- --help
```

Example:

```bash
cargo run -- init --project demo
```

This is useful while developing, but installing locally is more convenient for
real workflow testing.

## Install Locally

Install EnvGate from this checkout:

```bash
cd /Users/eduardonicoleit/Documents/env-gate
./install.sh
```

Verify the installed binary:

```bash
which envgate
envgate --help
```

The installer places the binary at `~/.local/bin/envgate`. If that directory is
not on your shell path, the installer prints one short fix:

```bash
echo 'export PATH="$HOME/.local/bin:$PATH"' >> ~/.zshrc
source ~/.zshrc
```

Once the GitHub repository slug is configured, the same installer can download a
tagged release instead of building locally:

```bash
ENVGATE_GITHUB_REPO=owner/envgate ./install.sh
ENVGATE_GITHUB_REPO=owner/envgate ENVGATE_VERSION=v0.1.0 ./install.sh
```

Without `ENVGATE_GITHUB_REPO`, `./install.sh` builds the local checkout with
Cargo.

## Safe Demo Flow

Use a temporary folder first so you do not touch a real project:

```bash
mkdir -p /tmp/envgate-demo
cd /tmp/envgate-demo

cat > .env <<'EOF'
DATABASE_URL=postgres://user:pass@localhost:5432/demo
PAYLOAD_SECRET=fake-payload-secret
NEXT_PUBLIC_API_URL=http://localhost:3000
EOF

cat > .gitignore <<'EOF'
.env
.env.*
!.env.vault
EOF
```

Run guided init:

```bash
envgate init --project demo
```

This creates `.envgate.json`, imports `.env` into `.env.vault`, verifies the
vault can decrypt, replaces plaintext `.env` with an EnvGate locked marker,
creates the initial run unlock session, registers the project, updates
`.gitignore`, and writes profile-based agent instructions to `AGENTS.md` unless
a `CLAUDE.md` file already exists.

The initial unlock starts the on-demand local broker. The broker keeps active
unlock/signing capability in memory and listens on
`~/.envgate/run/envgate.sock`. It is not an installed daemon and it does not
watch files or scan terminal input.

For scripts and CI-style local tests, the equivalent non-interactive entry point
is still available:

```bash
envgate setup --yes --project demo
```

Check the setup:

```bash
envgate doctor
```

Setup creates `dev` and `migrate` profiles from the detected package manager and
exact env names found in the verified vault. The generated commands default to
`pnpm dev` and `pnpm payload migrate` when no package manager is detected.
Generated profiles include only keys that exist in `.env.vault`; for example,
a project with `DATABASE_URI` but no `DATABASE_URL` will not receive a
`DATABASE_URL` profile entry.

Profiles live in `.envgate.json`. They intentionally store exact env names, not
wildcards, so an agent can ask for `--profile dev` without reading decrypted
vault contents:

```json
{
  "profiles": {
    "dev": {
      "command": "pnpm dev",
      "env": ["DATABASE_URL", "PAYLOAD_SECRET"],
      "defaultScope": "always",
      "action": "Run local development server"
    },
    "migrate": {
      "command": "pnpm payload migrate",
      "env": ["DATABASE_URL", "PAYLOAD_SECRET"],
      "defaultScope": "branch",
      "action": "Run database migration"
    }
  }
}
```

New `.envgate.json` files keep profiles as the normal user and agent command
layer. Presets are still supported for lower-level raw-command policy matching,
but they are advanced rules and are not generated by default.
Profile env membership is treated as the normal scope declaration, so routine
profile requests should not produce `env.scope_deviation` warnings.

## Run a Command With Scoped Secrets

Run a command with only `DATABASE_URL` injected:

```bash
envgate run \
  --agent codex \
  --action "Check database URL availability" \
  --env DATABASE_URL \
  -- sh -c 'test -n "$DATABASE_URL"'
```

All EnvGate flags must be before `--`. Everything after `--` is the child
command and its arguments:

```bash
# Correct
envgate run --agent codex --action "Run dev" --env DATABASE_URL --json --no-prompt -- pnpm dev

# Wrong: these flags go to pnpm, not EnvGate
envgate run --agent codex --action "Run dev" --env DATABASE_URL -- pnpm dev --json --no-prompt
```

Expected behavior:

- EnvGate asks for approval.
- EnvGate asks for the vault PIN/passphrase if no broker unlock session is active.
- The child command receives only the approved env vars.
- Encrypted audit logs are written under `~/.envgate/logs/`.

If a command appears to print or inspect secrets directly, EnvGate treats it as
critical and requires an explicit once-only confirmation:

```bash
envgate run \
  --agent codex \
  --action "Debug one env value" \
  --env DATABASE_URL \
  -- sh -c 'echo "DATABASE_URL=$DATABASE_URL"'
```

Critical commands print a prominent warning. The choices are restricted to
`Deny` and `Allow once`; durable `session`, `branch`, and `always` grants are not
allowed for critical findings.

To reduce prompts for a trusted dev command, create an allow grant for the
generated `dev` profile. Setup has already created the first run unlock session:

```bash
envgate allow --profile dev --scope always --agent codex
envgate dev --agent codex
```

The grant authorizes only the matching command/env scope. The unlock session is
still required before EnvGate can decrypt and inject secrets; refresh it later
with `envgate unlock --ttl 8h` after expiry or after `envgate lock`.

You can also run a profile explicitly:

```bash
envgate run --profile dev --agent codex --worktree "$PWD" --git-remote "$(git config --get remote.origin.url)" --commit "$(git rev-parse HEAD)" --branch "$(git branch --show-current)" --json --no-prompt
envgate run --profile migrate --agent codex --branch feature/migration
```

For agent no-prompt flows, the agent should always send full context rather
than waiting for EnvGate to ask: `--agent`, `--worktree`, `--branch`,
`--git-remote`, `--commit`, `--action`, and either `--profile` or exact
`--command` plus exact `--env` names. EnvGate independently verifies this local
Git/worktree context before approving or running.
If the repository has no `origin` remote, pass `--git-remote ""` explicitly.

Broker and worktree helpers:

```bash
envgate broker status
envgate broker socket-path
envgate worktrees list --project demo
envgate worktrees allow-root --project demo /tmp
```

Automatic worktree env delivery means scoped process env injection through
`envgate run`. It does not write plaintext `.env` files for agents.

## Manual Local Development

If you want to run a tool directly without an agent, unlock a plaintext `.env`
for yourself:

```bash
envgate env unlock
pnpm dev
envgate env lock
```

`envgate env unlock` writes plaintext `.env` with a warning header. `envgate env
lock` parses the current `.env`, re-encrypts `.env.vault`, verifies it can
decrypt, and restores the locked marker file.

Global project/env helpers:

```bash
envgate projects list
envgate projects show my-project
envgate projects use my-project
envgate env list --project my-project
envgate env set --project my-project OPENAI_API_KEY=sk-local
envgate env unset --project my-project OPENAI_API_KEY
envgate env export --project my-project --output .env.export
```

## Request Access Before Running

Use `envgate request` to create an approval grant without running a command:

```bash
envgate request \
  --profile dev \
  --agent claude \
  --worktree "$PWD" \
  --git-remote "$(git config --get remote.origin.url)" \
  --commit "$(git rev-parse HEAD)" \
  --branch "$(git branch --show-current)" \
  --json \
  --no-prompt
```

After the user approves in the agent UI:

```bash
envgate unlock --ttl 8h
envgate approve <request-id> --scope session --agent-mediated --json
```

`envgate approve` and `envgate allow` create signed approval receipts, so they
need an active broker unlock session. The unlock gives EnvGate temporary
broker-held signing capability; agents never receive the PIN/passphrase or
private key. If the agent sees `signing_key_unavailable` or
`"status": "unlock_required"`, ask the human to run `envgate unlock --ttl 8h`
and then retry the approval.

Agents should use native structured choice UI for `approvalOptions` when their
interface supports it. They must show `findings` to the user before approval,
especially `action.*` findings, because those indicate suspicious declared
action text.

For agent-mediated requests, always inspect the JSON response. If it contains
`"confirmationRequired": true`, the agent must show the `confirmation` warning
to the user and request explicit permission. Critical approvals are once-only:

```bash
envgate deny <request-id> --agent-mediated --json
envgate approve <request-id> --scope once --confirm-critical --agent-mediated --json
```

EnvGate refuses critical approvals without `--confirm-critical`, and it refuses
critical `session`, `branch`, or `always` approvals.

To deny:

```bash
envgate deny <request-id> --agent-mediated --json
```

Manual requests still work when no profile fits:

```bash
envgate request \
  --agent codex \
  --branch test-branch \
  --action "Allow DB for demo command" \
  --command "sh -c echo" \
  --env DATABASE_URL
```

Persisted grants can be:

- `session`: expires after 8 hours.
- `branch`: applies to the same project, command, env subset, and branch.
- `always`: applies to the same project, command, and env subset.

Grants skip future approval prompts only. EnvGate still needs an active run
unlock session or a fresh PIN/passphrase when `envgate run` injects secrets.

## Edit the Vault

Edit encrypted env contents safely:

```bash
envgate edit
```

EnvGate decrypts to a temporary file, opens `$EDITOR` or `$VISUAL`, validates
dotenv syntax, re-encrypts `.env.vault`, and removes the temporary file.

Set an editor if needed:

```bash
export EDITOR="nano"
```

For VS Code:

```bash
export EDITOR="code --wait"
```

## Validate, Lock, and Inspect Logs

Create a short-lived unlock session:

```bash
envgate unlock --ttl 8h
```

Revoke session-scoped grants and unlock sessions:

```bash
envgate lock
```

Print log locations:

```bash
envgate logs
envgate logs requests
envgate logs approvals
envgate logs executions
envgate logs alerts
envgate logs sessions
```

View encrypted logs after explicit approval/PIN/passphrase:

```bash
envgate logs view executions
envgate logs verify
envgate logs verify --full
envgate logs export executions --output executions.log
```

`envgate logs view` and `envgate logs export` always prompt for the vault
PIN/passphrase. `envgate logs verify` checks the cleartext hash-chain metadata
without decrypting payloads; `--full` also decrypts payloads after
PIN/passphrase validation. The encrypted log key is a local private file at
`~/.envgate/cache/log-key.json`; normal log access does not use the OS Keychain.

## Use EnvGate in a Real Project

From the project directory:

```bash
envgate init --project my-project
envgate doctor
```

Then replace secret-bearing commands with `envgate run`:

```bash
envgate allow --profile dev --scope always --agent codex
envgate dev --agent codex
```

## Multi-Worktree Usage

Register the canonical project once:

```bash
cd /path/to/main/project
envgate register my-project
```

From a worktree or temporary clone, EnvGate can resolve the same project through
local config, git remote, path ancestry, or active project selection:

```bash
envgate use my-project
envgate run --profile migrate --agent codex --branch feature/migration
```

## Reset Local Test State

For a demo project:

```bash
rm -rf /tmp/envgate-demo
```

To remove EnvGate's local registry, grants, and logs:

```bash
rm -rf ~/.envgate
```

To remove EnvGate from one project while preserving logs:

```bash
envgate teardown --yes
```

By default teardown exports plaintext secrets to `.env.export`, not `.env`. To
explicitly restore plaintext `.env` during teardown:

```bash
envgate teardown --yes --restore-env
```

To uninstall the local binary:

```bash
rm -f ~/.local/bin/envgate
```

## Important Limits

EnvGate does not protect secrets if you bypass it:

```bash
cat .env
printenv
pnpm dev
```

The intended passive MVP workflow is:

1. Encrypt `.env` into `.env.vault`.
2. Keep `.env` locked unless you explicitly run `envgate env unlock`.
3. Run secret-bearing commands through `envgate run`.
4. Review encrypted logs through PIN/passphrase-gated `envgate logs view` when needed.

EnvGate is not anti-malware. A same-user process can still delete local files if
the OS allows it. Encrypted hash-chained logs provide confidentiality and tamper
evidence, not undeletable audit storage.

Approval grants are also tamper-evident. Each reusable grant includes a signed
receipt for the exact project, agent, branch, command hash, env names, scope,
expiry, request id, and critical-confirmation state. Edited or legacy unsigned
grants are ignored and should be recreated through the normal approval flow.
